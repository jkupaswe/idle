# Idle — Product Requirements Document

**Version:** 0.1 (v1 scope)
**Target:** shippable in 1-2 hours of parallel agent work
**Stack:** Node 20+, TypeScript, distributed via npm

---

## 1. What Idle is

Idle is a command-line tool that integrates with Claude Code via its hook system to give developers gentle, deterministic check-ins during long agentic coding sessions. When tool-use or time thresholds trip, Idle surfaces a short, context-aware break suggestion generated by Claude itself (using the user's existing Claude Code authentication — no API keys, no proxying, no inference cost to Idle).

The core thesis: managing multiple coding agents in parallel creates a novel form of cognitive fatigue (supervisory load from parallel asynchronous delegation) that generic Pomodoro tools don't address. Idle meters the *actual* signal — tool calls, session duration, and when possible token usage — and intervenes at natural break points (agent stop events), not on arbitrary timers.

## 2. What Idle is not

- Not a wellness app. No streaks, no confetti, no gamification, no mood tracking.
- Not a GUI. v1 is terminal and native OS notifications only.
- Not a timer. Timers are a commodity and not the product.
- Not paternalistic. The user can dismiss, disable per-project, or kill-switch in one command at any time.
- Not cloud-dependent. Everything runs locally except the LLM call for break suggestions, which uses the user's own Claude Code auth.
- Not a Claude Code replacement or wrapper. It's a companion that installs into the user's existing Claude Code setup via standard hook configuration.

## 3. Target user

Individual software engineer using Claude Code, typically running one or more long agentic sessions per day. Terminal-native. Likely already has 5-20 MCP servers or hooks configured. Skews toward indie developers, founders, and developer-tool enthusiasts (the HN-reading audience). Does not want to be told what to do; does want a tool that earns their trust by being useful and unobtrusive.

## 4. Voice and tone (product-level)

The product has a voice. All user-facing copy (CLI output, README, help text, break suggestions) follows these rules:

- Dry, matter-of-fact, occasionally witty. Never cheerful. Never preachy.
- Unix-tool aesthetic. Think `htop`, `fd`, `ripgrep`, `bat` — not Duolingo.
- Short. Every sentence earns its place.
- Self-aware about what the tool is. "A break timer that meters your tokens, not your minutes" is on-brand. "Take care of yourself today! 💚" is not.
- No emoji in output by default. If a user configures an emoji-enabled tone, use them sparingly.
- Honest about limits. If a feature is experimental or best-effort, say so.

## 5. User journey (v1)

1. User runs `npx idle init` (or `npm install -g idle && idle init`).
2. Interactive TUI walks them through: thresholds (time + tool calls), tone preset, per-project enable/disable. ~30 seconds.
3. Installer writes Idle's hooks into `~/.claude/settings.json` (merging, not clobbering) and creates `~/.idle/` with the config and empty state.
4. User starts a Claude Code session. Idle is active.
5. As the session proceeds, `PostToolUse` hooks quietly increment counters in `~/.idle/state.json`. No output. Zero latency.
6. When a threshold trips AND the agent hits a natural break (Stop event), a native OS notification appears with a single-sentence break suggestion, generated by a `prompt`-type hook using the user's own Claude Code auth. Example: "You've been at this 47 minutes and 32 tool calls. Walk to the window and look at something more than 20 feet away for a minute."
7. The user can acknowledge, dismiss, or ignore. Idle resets the threshold counter and continues.
8. `idle stats` at any time shows current session stats. `idle disable` and `idle enable` toggle per-project. `idle uninstall` removes everything cleanly.

## 6. Functional requirements

### 6.1 Installer (`idle init`, `idle install`, `idle uninstall`)

- `idle init` runs interactive setup. Writes config to `~/.idle/config.toml`. Adds hook entries to `~/.claude/settings.json` using a deep-merge that preserves all existing user hooks.
- `idle install` is non-interactive equivalent, usable from scripts. Reads config from flags or uses defaults.
- `idle uninstall` removes Idle's hook entries from `~/.claude/settings.json` (identified by a tag in the command string). Preserves `~/.idle/` config by default; `--purge` flag removes everything.
- All file writes to `~/.claude/settings.json` must be atomic (write to temp file, then rename) and must create a backup at `~/.claude/settings.json.idle-backup-<timestamp>` before modifying.
- Installer refuses to run if Claude Code is not installed (checks for `~/.claude/` directory and `claude` binary on PATH).

### 6.2 Config (`~/.idle/config.toml`)

Minimum schema:

```toml
[thresholds]
time_minutes = 45          # elapsed time since last check-in
tool_calls = 40            # tool calls since last check-in
# Either threshold trips the check-in. Set to 0 to disable that threshold.

[tone]
preset = "dry"             # one of: "dry", "earnest", "absurdist", "silent"
# "silent" suppresses the LLM-generated suggestion; shows stats only.

[notifications]
method = "native"          # "native" (OS notification), "terminal" (stderr print), "both"
sound = false

[projects]
# Per-project overrides. Key is absolute path.
# "/Users/justin/some-grind-project" = { enabled = false }
```

### 6.3 State (`~/.idle/state.json`)

Tracked per Claude Code session (identified by session ID from hook input):

```json
{
  "sessions": {
    "<session_id>": {
      "started_at": "<ISO timestamp>",
      "project_path": "<absolute path>",
      "tool_calls_since_checkin": <int>,
      "total_tool_calls": <int>,
      "last_checkin_at": "<ISO timestamp | null>",
      "checkins": [<array of timestamps>]
    }
  }
}
```

State file must be read and written atomically (flock or write-temp-then-rename) because multiple hooks may fire concurrently.

### 6.4 Hooks

Four hooks, all registered under matcher `""` (universal) unless specified:

**SessionStart** (command, async):
- Initializes session entry in state.json.
- Records project_path from `$CLAUDE_PROJECT_DIR`.
- Checks if project is disabled in config; if so, writes a `{"disabled": true}` flag into the session entry so other hooks short-circuit.

**PostToolUse** (command, async):
- Increments `tool_calls_since_checkin` and `total_tool_calls` for the session.
- Writes a "pending_checkin" flag to the session entry if either threshold is crossed. Does NOT trigger the notification (too noisy mid-tool-use).

**Stop** (prompt):
- Fires when the agent finishes responding.
- Reads session state. If `pending_checkin == true`, invokes the prompt handler.
- Prompt template (parameterized by tone):

```
The developer has been in a Claude Code session for {duration_minutes} minutes and the agent has performed {tool_calls} tool calls since the last check-in. The last tool was `{last_tool_name}` operating on `{last_tool_summary}`.

Write a single-sentence break suggestion in a {tone} voice. Be specific to the work when relevant. No hedging. No wellness-app language. Max 30 words. Return only the sentence, no preamble, no quotes.
```

- The prompt output is passed to a small shell command that triggers a native notification (`osascript -e 'display notification ...'` on macOS, `notify-send` on Linux).
- After notification fires, reset `tool_calls_since_checkin` to 0 and `last_checkin_at` to now.

**SessionEnd** (command, async):
- Writes session summary to `~/.idle/sessions/<session_id>.json` for optional later review.
- Removes session entry from live state.json.

### 6.5 CLI commands

- `idle init` — interactive setup
- `idle install [--defaults]` — non-interactive install
- `idle uninstall [--purge]` — remove hooks
- `idle stats [--session <id> | --today | --all]` — show session stats
- `idle disable` — disable for current project (writes to config based on `pwd`)
- `idle enable` — re-enable for current project
- `idle status` — show whether Idle is active, installed, and enabled for this directory
- `idle doctor` — diagnostic check (hooks registered correctly, Claude Code detected, state file readable, etc.)

### 6.6 Tone presets

Four presets, each with a distinct voice baked into the prompt template:

- **dry** (default): matter-of-fact, occasionally observational. "Forty minutes in. Stand up, look at something more than ten feet away."
- **earnest**: direct but warmer, slightly more encouraging. "You've been at this for a while — a short walk would help you come back sharper."
- **absurdist**: dry with an odd specific suggestion. "Go put three unrelated objects in a line on your desk. Come back."
- **silent**: no LLM call; just a notification with bare stats. "45m / 42 tool calls."

Presets are implemented as different prompt templates in `src/prompts/`, selected by config.

## 7. Non-functional requirements

- **Zero token cost at steady state.** The only LLM call is the Stop-hook prompt, and only when a threshold is tripped. Typical session should have 2-5 LLM calls max per hour.
- **Zero added latency to Claude Code tool use.** All PostToolUse work is async and takes <50ms to complete.
- **No dependencies on external services.** Idle does not phone home. No telemetry. No analytics. No network calls except the user-auth'd Claude Code prompt call.
- **Cross-platform.** macOS and Linux in v1. Windows can be v2.
- **Reversible installation.** `idle uninstall` returns the system to exactly the state before `idle init`, with a backed-up settings.json to prove it.
- **Well-behaved citizen.** Never modifies anything under `~/.claude/` except `settings.json` (and only the keys Idle owns). Never modifies the user's git state, shell config, or other dev tools.

## 9. Installation flow detail

When `idle install` runs:

1. Resolve home dir, confirm `~/.claude/` exists and `claude` is on PATH.
2. Read existing `~/.claude/settings.json` (or create empty object if missing).
3. Write backup to `~/.claude/settings.json.idle-backup-<timestamp>`.
4. Merge Idle hook entries into the `hooks` key. Each Idle hook is marked with a comment-style tag in its command string, e.g. `npx tsx <path>/session-start.ts # idle:v1`, so uninstall can find and remove them deterministically.
5. Create `~/.idle/config.toml` from template (or interactive answers).
6. Create `~/.idle/state.json` as `{"sessions": {}}`.
7. Create `~/.idle/sessions/` directory for session summaries.
8. Create `~/.idle/debug.log` (empty).
9. Print success with `idle status` output.

## 10. Acceptance criteria for v1 ship

- `npx idle init` walks a new user through setup in under 60 seconds and results in a working install.
- After install, starting a fresh `claude` session and running any agentic task for >45 min results in a native notification with a tone-appropriate break suggestion.
- `idle stats --today` shows accurate session count, total tool calls, and check-in count.
- `idle uninstall` restores `~/.claude/settings.json` to its pre-install state (verified by diff against backup).
- `idle doctor` correctly reports a green status on a working install and clear diagnostics on a broken one.
- The install does not break any other Claude Code hooks or functionality for the user.
- README clearly explains install, tone options, privacy posture (no telemetry, no API keys), and uninstall.
- Test suite covers: settings.json merge/unmerge, config load/save, state atomic writes, each hook script with synthetic stdin fixtures, CLI command smoke tests.

## 11. Out of scope for v1

- Windows support
- Token-based metering (requires parsing Claude Code session files, not exposed in hook payloads yet — defer until Anthropic exposes it, or V2 with file-tailing)
- GUI config editor
- Custom tone (user-authored prompt templates) — v2
- Per-agent-team breakdowns when Agent Teams are in use — v2
- Analytics dashboard — v3 if ever
- Team/org features — probably never, this is a single-player tool

## 12. Launch plan (informational — not part of the build)

v1 ships open-source under MIT. README is the marketing. HN post headline candidate: "Idle: a break timer that meters your tokens, not your minutes." Success metric for the launch experiment is GitHub stars + npm downloads at 7 days post-HN post. If it pops, build v2 with token metering and a richer config system. If it doesn't, it's still a useful tool for Justin and ~100 others.
